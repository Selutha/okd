## Map vs the original cidr-plan.md slices

Original plan claimed eight clean /19-pair slots. Reality:

| Slot range | Plan said | Reality |
|---|---|---|
| 192.168.0.0–31.255 | mgmt cluster (pods + svc) | **Service CIDR overlaps hpg-roce (20.0/24) and hpg-vmotion (30.0/24)** |
| 192.168.32.0–63.255 | reserved | **Saturated with switch mgmt + ResVault subnets** |
| 192.168.64.0–95.255 | reserved | **Consumed entirely by ResVault Security /18** |
| 192.168.96.0–127.255 | reserved | **Consumed entirely by ResVault Security /18** |
| 192.168.128.0–159.255 | reserved | Pods half (128–143) used by hpgai-compute; services half (144–159) appears free |
| 192.168.160.0–191.255 | reserved | Pods half (160–175) appears free; services half (176–191) used by hpg2-drbd |
| 192.168.192.0–223.255 | reserved | **Both halves used by hpg2-corosync-1 and -2** |
| 192.168.224.0–255.255 | reserved | Pods half (224–239) used by hpgovx-compute; services half (240–255) appears free |

## What's actually free in 192.168/16 (clean /20s)

Three /20 blocks with no listed conflicts:
- **192.168.144.0/20** (16 /24s)
- **192.168.160.0/20** (16 /24s)
- **192.168.240.0/20** (16 /24s)

That's enough for one cluster (one /20 pods + one /20 svc) plus a single spare /20. Not enough for the planned 8-cluster fleet.

## Implications

- The "192.168/16 is K8s home" invariant from the original cidr-plan.md is dead.
- Re-rolling the mgmt cluster is necessary — its current service CIDR (192.168.16.0/20) overlaps the FA storage subnet, which will bite the moment K8s nodes get a path to the storage network.
- For fleet-scale (multiple workload clusters), 192.168/16 is too fragmented. Either pivot to 100.64.0.0/10 (existing notes in `cidr-100.64-notes.md`) or carve out a clean /16 from 172.16/12 with the network team.
